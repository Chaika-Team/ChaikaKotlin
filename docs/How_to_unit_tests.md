# Руководство по Unit‑тестам в нашем Android‑проекте (Kotlin + Hilt + Flow)

**Цель документа:** дать единые правила и практики, чтобы unit‑тесты приносили реальную пользу (ловили регрессии в бизнес‑логике), были быстрыми, детерминированными и не ломались при каждом рефакторинге.

> Мы пишем тесты не ради процента покрытия, а ради защиты инвариантов продукта и данных.

---

## 0) Что у нас уже подключено (по текущему build.gradle.kts)

### Основной стек unit‑тестов (`src/test`)
- **JUnit 5** (запуск через `useJUnitPlatform()` + плагин `de.mannodermaus.android-junit5`).
- **Kotest 5.9.1** (runner + assertions + property).
- **MockK 1.13.11** (моки для Kotlin + корутин).
- **kotlinx-coroutines-test 1.8.1** (`runTest`, тестовые диспетчеры).
- **Turbine 1.1.0** (удобное тестирование `Flow`).
- **Robolectric 4.14** (+ robolectric-extension для JUnit5) — для JVM‑тестов, которым нужен Android framework/Room/Resources.
- **Paging testing** (`paging-testing`).
- **Jacoco** (отчёты покрытия).

⚠️ В Gradle есть и **Mockito** одновременно с **MockK**. Для здоровья проекта **выбираем один мок-фреймворк** как основной (см. раздел 2.3).

## 1) Термины и уровни тестов

### 1.1 Unit‑тест (JVM)
**Unit** = тестирует бизнес‑логику в памяти, без реальной сети и без реальной БД.
- Быстрый (миллисекунды).
- Детерминированный.
- Работает в `src/test`.

Типичные кандидаты:
- use case’ы (ветвления, статусы, правила, идемпотентность);
- мапперы и валидации;
- маппинг сетевых результатов → доменные результаты;
- логика ViewModel, если она не требует Android runtime.

### 1.2 Интеграционный JVM‑тест (Robolectric)
Тест на JVM, но использует Android‑части: Room, SharedPreferences, ресурсы.
- Может быть чуть медленнее.
- Всё ещё `src/test`, но с Robolectric.

Типичные кандидаты:
- DAO/SQL запросы Room (фильтры, сортировка, индексы, транзакции);
- репозитории поверх Room;
- сериализация/десериализация (Moshi) при необходимости.

### 1.3 Инструментальный тест (`src/androidTest`)
Запускается на устройстве/эмуляторе.
- Дороже и медленнее.
- Пишем минимально: только критические end‑to‑end сценарии.

## 2) Стандарты по библиотекам и стилю

### 2.1 Kotest или JUnit?
Мы используем **Kotest** для большинства новых unit‑тестов:
- удобный DSL (`FunSpec`, `test("...") { }`);
- читабельные ассерты (`shouldBe`);
- data-driven/property-based там, где это реально полезно.

**Допустим JUnit5**, если:
- тест проще/короче в JUnit;
- тест использует специфические JUnit extensions;
- или нужно быстро добавить маленький тест без DSL.

### 2.2 Коррутины
**Всегда** используем:
- `kotlinx.coroutines.test.runTest { ... }`

Запрещено в unit‑тестах:
- `delay(...)` для ожиданий “на удачу”;
- реальные `Dispatchers.IO/Default/Main` без контроля.

Если код зависит от диспетчера — используем `DispatcherProvider` (или аналог) и подменяем в тестах.

### 2.3 Моки: MockK vs Mockito
**Рекомендация проекта:** использовать **MockK** как основной.

Почему:
- Kotlin‑friendly;
- `coEvery/coVerify` для `suspend` функций;
- меньше “магии” и конфликтов с final‑классами.

**Важно:** не мешаем MockK и Mockito в одном пакете тестов без причины.
- Если исторически есть Mockito‑тесты — можно временно оставить.
- Все новые тесты пишем на **MockK**.

### 2.4 Flow
Для тестирования Flow/StateFlow/SharedFlow используем **Turbine**:
- `flow.test { awaitItem(); ... }`

## 3) Что именно тестировать в нашем проекте

### 3.1 Приоритет 1 (обязательно)
1) **Use case’ы** (смены, отчёты, отправка, статусы):
   - ветвления по статусам;
   - идемпотентность (повторный вызов не ломает состояние);
   - корректные side‑effects (что должно обновиться/не должно).

2) **Маппинг результатов сети** (пример: `UploadShiftReportUseCase`):
   - `Ok → Success`
   - `Http 409 → Success`
   - `4xx → PermanentFailure`
   - `5xx → TemporaryFailure`
   - `NetworkError → TemporaryFailure(isNetwork=true)`

3) **Критичные мапперы/валидаторы**, которые легко сломать:
   - статус <-> int код;
   - сериализация отчётов (если формат критичен).

### 3.2 Приоритет 2 (желательно)
- ViewModel (если много логики и преобразований Flow);
- сценарии “ошибка → повтор → успех”;
- мелкие helpers, если они содержат условия.

### 3.3 Обычно НЕ тестируем unit‑тестами
- простые data‑классы без логики;
- DI‑модули Hilt (как правило, достаточно компиляции + 1 smoke интеграционный тест);
- тонкие прокладки “один метод просто делегирует в repo” — если там нет условий.

> Если в прокладке появляется условие/ветвление — это уже кандидат на unit‑тест.

## 4) Главный принцип: тестируем поведение и инварианты

**Хороший тест отвечает на вопрос:**
> “Какое правило продукта защищает этот тест?”

Примеры инвариантов:
- «Если смена SENT — не отправлять повторно».
- «Если отчёта нет — не ходить в сеть».
- «Помечать SENT только после успешной отправки».

**Плохой тест**:
- проверяет “вызвали метод репозитория ровно 1 раз” без бизнес‑смысла;
- ломается при любом рефакторинге, хотя поведение не изменилось.

## 5) Техники тест‑дизайна (используем осознанно)

Ниже — техники, которые реально помогают нам писать “умные” тесты:

1) **Классы эквивалентности**
   - Группируем входы/состояния на классы, где поведение одинаково.
   - Пример: статус `SENT` — отдельный класс; `FINISHED+report` — отдельный класс.

2) **Граничные значения**
   - Проверяем границы диапазонов/условий.
   - Пример: `reportJson == null/blank`; HTTP код 409 как исключение; 499/500 как граница.

3) **Таблица решений (Decision Table)**
   - Составляем комбинации условий → результат.
   - Отлично подходит для use case’ов со статусами/ошибками.

4) **Переходы состояний (State Transition)**
   - Используем, когда есть жизненный цикл сущности.
   - Пример: `FINISHED → SENT` только при `Success`.

5) **Error Guessing / Анализ типичных ошибок**
   - Добавляем тесты под самые вероятные регрессии:
     - перепутали статус;
     - начали ставить SENT при TemporaryFailure;
     - забыли обработать 409.

> Технику указываем в комментарии к тесту (как в ваших примерах) — это дисциплинирует и помогает ревью.

## 6) Чеклист: полезный ли тест?

Перед тем как принять тест в PR, проверь:

### 6.1 Бизнес‑ценность
- Какой инвариант/правило продукта защищает тест?
- Что может сломаться у пользователя/в данных, если этого теста нет?

### 6.2 Реалистичность
- Этот сценарий реально возможен в проде?
- Это типичный класс ошибок/падений?

### 6.3 Поведение vs реализация
- Тест проверяет результат/сайд‑эффект, а не внутренние детали.
- `verify` используем только для важных внешних эффектов.

### 6.4 Детерминированность
- Нет `delay()`.
- Используется `runTest`.
- Для Flow — Turbine.

### 6.5 Устойчивость к рефакторингу
- Не проверяем порядок вызовов, если порядок не является требованием.
- `confirmVerified(...)` используем экономно (иначе тесты хрупкие).

### 6.6 Читаемость
- Название теста отражает правило.
- Внутри есть структура Given/When/Then.
- Минимум шума и дубликатов.

## 7) Шаблоны оформления тестов

### 7.1 Базовый шаблон Kotest + MockK + coroutines
```kotlin
class ExampleUseCaseTest : FunSpec({

  lateinit var dep: SomeDependency
  lateinit var useCase: SomeUseCase

  beforeTest {
    dep = mockk()
    useCase = SomeUseCase(dep)
  }

  test("when ... - then ...") {
    runTest {
      // Given
      coEvery { dep.call(...) } returns ...

      // When
      val result = useCase(...)

      // Then
      result shouldBe ...
      coVerify(exactly = 1) { dep.call(...) }
    }
  }
})
```

### 7.2 Когда использовать `coVerifyOrder`
Только если порядок — бизнес‑инвариант.
Пример: «помечаем SENT только после успешного upload».

### 7.3 Captor аргументов (MockK)
```kotlin
val slot = slot<SomeType>()
coVerify { repo.save(capture(slot)) }
slot.captured shouldBe expected
```

## 8) Flow‑тесты через Turbine (рекомендуемый способ)

Если use case/VM возвращает Flow:
```kotlin
someFlow.test {
  awaitItem() shouldBe ...
  cancelAndIgnoreRemainingEvents()
}
```

**Не используем**: `delay()` + ручные коллекции.

## 9) Hilt и тесты

### 9.1 Для unit‑тестов (рекомендуется)
Hilt в unit‑тестах **обычно не нужен**.
- Просто создаём use case и подсовываем моки/фейки.

### 9.2 Для интеграционных/инструментальных
Используем `@HiltAndroidTest` + `HiltAndroidRule`, если нужно тестировать wiring зависимостей.

---

## 10) Room/DAO: где тестировать

- Логику SQL/индексов/транзакций лучше тестировать **интеграционно**:
  - либо Robolectric (JVM),
  - либо `androidTest` на устройстве.

Примеры того, что важно проверить:
- сортировка истории смен;
- фильтр “всё кроме ACTIVE”;
- гарантия “не более одной ACTIVE” (через индекс/ограничение);
- транзакции “сформировал отчёт → записал → очистил операции”.

---

## 11) Покрытие (Jacoco) и здравый смысл

Jacoco у нас подключён — отчёт полезен, но:
- **не гонимся за % ради %**;
- целимся в покрытие **домена и бизнес‑веток**;
- UI/DI/платформенный код покрывать процентом бессмысленно.

Рекомендуемая практика:
- каждый баг → тест, который воспроизводит баг (красный на старом коде, зелёный на новом).

---

## 12) Мини‑памятка: как писать тесты быстро и правильно

1) Сначала выпиши **матрицу решений** для use case (условия → результат).
2) Выбери 5–10 самых важных строк матрицы и напиши на них тесты.
3) Убедись, что каждый тест защищает **инвариант**.
4) Избегай хрупкости: минимум verify‑шумов, порядок только по необходимости.
5) Тесты должны быть быстрыми и детерминированными.

## Приложение: пояснение MockK‑синтаксиса (коротко)

- `coEvery { ... } returns ...` — настроить поведение мока для `suspend` функции.
- `coVerify(exactly = N) { ... }` — проверить, сколько раз вызвали `suspend` функцию.
- `coVerifyOrder { ... }` — проверить порядок вызовов (использовать редко).
- `confirmVerified(mock)` — убедиться, что не было других вызовов (использовать экономно).

---

### Что делать, если тесты “постоянно ломаются”
Почти всегда причина одна из этих:
- тесты проверяют детали реализации (лишние `verifyOrder/confirmVerified`);
- тесты зависят от времени/диспетчеров;
- тесты лезут в платформу вместо unit‑границы.

Фикс: возвращаемся к инвариантам и матрице решений.
